1. Обрані лінтери та причини вибору
   Для забезпечення надійності математичного рушія fuzzy-logic, який є критичним компонентом Backend-сервера системи моніторингу мікроклімату, було обрано комбінований підхід з використанням двох інструментів статичного аналізу: Checkstyle та PMD.

Причини вибору:

Розподіл обов'язків на рівні абстракцій: Checkstyle працює виключно з вихідним текстом коду, автоматизуючи перевірку на відповідність стандартам кодування, що звільняє час розробників на Code Review та суперечки щодо форматування. Натомість PMD будує абстрактне синтаксичне дерево (AST), що дозволяє знаходити глибокі структурні та логічні помилки (code smells), такі як надмірна складність або порожні блоки перехоплення виключень.

Апаратні та бізнес-обмеження: Бекенд системи (Java 21) безперервно обробляє дані з ESP32 шлюзів по REST API. Наявність "мертвого коду", надмірно складних алгоритмів або постійне створення непотрібних об'єктів призводить до перевантаження Garbage Collector'а. Обрані інструменти вміють виявляти такі вузькі місця до того, як вони потраплять у продакшен.

Інтеграція в CI/CD: Обидва інструменти мають офіційні плагіни для системи збірки Maven і можуть працювати на різних фазах життєвого циклу (validate для швидкого Checkstyle та виклик check для PMD), що дозволяє гнучко керувати ресурсами складальних серверів.

2. Базові правила та їх пояснення
   Правила Checkstyle
   Для перевірки стилю використовується конфігурація Google Java Style (google_checks.xml), яка є де-факто індустріальним стандартом і постачається разом з плагіном.

Форматування та іменування: Жорстко регламентує відступи, довжину рядків та іменування змінних (щоб зменшити когнітивне навантаження під час читання коду).

Ігнорування (Suppressions): Додано файл config/checkstyle/checkstyle-suppressions.xml. У ньому налаштовано винятки для тестових класів (.*Test\.java). У тестах дозволено відсутність Javadoc та використання "магічних чисел" (Magic Numbers) для зручності написання тестових сценаріїв.

Правила PMD
PMD налаштовано на перевірку трьох основних категорій: bestpractices.xml, design.xml та errorprone.xml. Найважливіші правила для нашого проєкту:

CyclomaticComplexity (з набору design): Обмежує кількість розгалужень (if, while, for) всередині одного методу. Математичні функції нечіткої логіки мають бути декомпозовані. Якщо складність занадто висока, алгоритм стає важким для підтримки та небезпечним для процесора сервера.

UseUtilityClass (з набору bestpractices): Вимагає, щоб класи, які містять лише статичні методи (наприклад, допоміжні математичні розрахунки), мали приватний конструктор. Це унеможливлює створення непотрібних екземплярів об'єктів у пам'яті.

UnusedPrivateField / UnusedLocalVariable: Знаходить змінні, які були оголошені, але ніколи не використовуються, що запобігає засміченню оперативної пам'яті.

3. Інструкція з запуску лінтера
   Лінтери повністю інтегровані у систему збірки Maven. Запуск можна здійснювати з термінала.

Основна команда для локального аудиту:
mvn clean validate checkstyle:check pmd:check

Пояснення до команди:

clean: Видаляє попередньо скомпільовані артефакти у папці target.

validate: Запускає перевірку maven-checkstyle-plugin. Якщо код не відповідає стилю Google, процес миттєво зупиняється з помилкою BUILD FAILURE.

checkstyle:check та pmd:check: Примусово генерують звіти для обох інструментів.

Аналіз результатів:

Якщо помилок немає, консоль виведе BUILD SUCCESS.

Якщо PMD знайде структурні помилки, збірка впаде, а в терміналі буде вказано конкретний файл, рядок та назву порушеного правила. Детальний XML-звіт про помилки від PMD можна знайти у файлі target/pmd.xml.

4. Інтеграція з процесом збірки та CI/CD
   Лінтинг повністю інтегровано у життєвий цикл збірки Maven. Комплексна перевірка коду, яка використовується розробниками та CI/CD сервером (наприклад, GitHub Actions), виконується єдиною командою:
   mvn clean verify
   Ця команда послідовно запускає перевірки на етапах validate, compile та verify, гарантуючи, що жоден артефакт не буде зібраний у разі наявності технічного боргу.

5. Захист репозиторію (Git Hooks)
   Для реалізації парадигми Shift-Left налаштовано pre-commit хук. Скрипт знаходиться у файлі .git/hooks/pre-commit. Його завдання — перехопити ініціативу до моменту створення коміту та запустити експрес-перевірку (Checkstyle) виключно для змінених .java файлів. Якщо виявлено порушення стандарту форматування, хук фізично блокує коміт, змушуючи розробника виправити код. Це економить ресурси CI-сервера та час на Code Review.

6. Розширена статична типізація
   Оскільки стандартний компілятор Java допускає певну свободу, що може призвести до логічних помилок у runtime, до процесу компіляції інтегровано плагін Google Error Prone. Він виконує глибокий статичний аналіз типів (Static Type Checking) прямо під час генерації байт-коду (через втручання в AST компілятора javac). Це дозволяє виявляти складні семантичні помилки, такі як некоректне порівняння масивів або проблеми багатопотоковості, трактуючи їх як помилки компіляції.
